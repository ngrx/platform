# V22 Update Guide

## Angular CLI update

NgRx supports using the Angular CLI `ng update` command to update your dependencies. Migration schematics are run to make the upgrade smoother. These schematics will fix some of the breaking changes.

To update your packages to the latest released version, run the command below.

```sh
ng update @ngrx/store@22.0.0-beta.0
```

## Dependencies

Version 22 has the minimum version requirements:

- Angular version 22
- Angular CLI version 22
- TypeScript version 6.0
- RxJS version ^6.5.x || ^7.5.0

## Breaking changes

### @ngrx/store

#### Enforce exact return types in `on()` callbacks

The `on()` callback return types are now strictly checked for excess
properties. Code that previously compiled with extra properties in
these callbacks will now produce TypeScript compilation errors with a
descriptive message.

When the state type is an unresolved generic parameter (e.g. a generic
reducer factory function), callbacks that spread state and override
known properties may produce a false type error. Return state directly
or use a type assertion (`as T`) in those cases.

BEFORE:

```ts
const reducer = createReducer(
  initialState,
  on(setName, (state, { name }) => ({
    ...state,
    name,
    extra: true, // no error
  }))
);
```

AFTER:

```ts
// Type '{ extra: boolean; ... }' is not assignable to type
// '"callback return type must exactly match the state type.
//  Remove excess properties."'
const reducer = createReducer(
  initialState,
  on(setName, (state, { name }) => ({
    ...state,
    name,
    extra: true, // TS error with descriptive message
  }))
);
```

### @ngrx/component-store

#### Enforce exact return types in `updater()` callbacks

The `updater()` callback return types are now strictly checked for
excess properties. Code that previously compiled with extra properties
in updater callbacks will now produce TypeScript compilation errors
with a descriptive message.

When the state type is an unresolved generic parameter (e.g.
`class MyStore<T> extends ComponentStore<T>`), callbacks that spread
state and override known properties may produce a false type error.
Return state directly or use a type assertion (`as T`) in those cases.

BEFORE:

```ts
this.updater((state, value: string) => ({
  ...state,
  name: value,
  extra: true, // no error
}));
```

AFTER:

```ts
// Type '{ extra: boolean; ... }' is not assignable to type
// '"updater callback return type must exactly match the state type.
//  Remove excess properties."'
this.updater((state, value: string) => ({
  ...state,
  name: value,
  extra: true, // TS error with descriptive message
}));
```

### @ngrx/eslint-plugin

#### Drop support for ESLint v8

BEFORE:

You could configure the NgRx plugin to use the ESLint v8 format.

AFTER:

Support for ESLint v8 is dropped, only the flat config syntax is
supported to register the NgRx plugin.

#### Remove `updater-explicit-return-type` and `on-function-explicit-return-type`

The `updater-explicit-return-type` and
`on-function-explicit-return-type` rules are removed.

BEFORE:

You could configure and run these rules in the NgRx ESLint plugin.

AFTER:

These rules are no longer available and should be removed from your
ESLint configuration.

### `@ngrx/operators`

#### Remove deprecated `tapResponse` signature

The deprecated callback-style signature of `tapResponse` is removed.

BEFORE:

The callback signature could be used.

```ts
tapResponse(
  (value) => {},
  (error) => {},
  () => {} // optional
);
```

AFTER:

Use the object-style signature.

```ts
tapResponse({
  next: (value) => {},
  error: (error) => {},
  complete: () => {}, // optional
});
```

### `@ngrx/entity`

#### Infer `selectId` return type from adapter configuration

BEFORE:

The `selectId` function in the entity adapter configuration has a
`string | number` return type.

AFTER:

The `selectId` function in the entity adapter configuration is inferred
from the adapter configuration. The return type is `string` or `number`,
depending on the implementation of the `selectId` function.
If no `selectId` function is provided, the default implementation infers
the return type from the `id` property of the entity.

BEFORE:

```ts
interface User {
  uid: string;
  name: string;
}

const adapter = createEntityAdapter<User>({
  selectId: (user) => user.uid,
});

// `id` is typed as string | number
adapter.getInitialState({
  selectedId: null as string | number | null,
});
```

AFTER:

```ts
interface User {
  uid: string;
  name: string;
}

const adapter = createEntityAdapter<User>({
  selectId: (user) => user.uid,
});

// `id` is typed as string (inferred from selectId)
const selectUserId = (user: User) => adapter.selectId(user);
```
